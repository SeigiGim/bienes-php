Épica única

Control digital del inventario de activos fijos

La épica raíz que agrupa todas las funcionalidades del sistema

EP-01 · Control digital del inventario de activos fijos
Objetivo de negocio: Reemplazar el control manual de activos (Excel + correos) por una API que permita registrar, ubicar, asignar y controlar el estado de cada bien físico de la organización en tiempo real, eliminando pérdidas de activos, compras duplicadas y la falta de trazabilidad sobre quién tiene qué y dónde.

Criterios de éxito — métricas de completitud

1
El 100% de los bienes puede ser registrado con etiqueta única. Cero duplicados de etiqueta posibles.
2
Cada bien tiene un estado definido (activo, en reparación, dado de baja) y un responsable o ubicación asignada.
3
Ningún bien se borra permanentemente: toda baja usa soft delete y es recuperable desde el historial.
4
La app móvil puede consultar, filtrar y actualizar bienes sin errores 500 en el 95% de los requests.
5
Los datos maestros (categorías, ubicaciones, responsables) se gestionan 100% desde la API, sin valores hardcodeados.
6
Todas las HU de prioridad Alta (MVP) pasan sus criterios de aceptación en ambiente de staging.

Feature F-01

Gestión de datos maestros

Categorías, ubicaciones y responsables — los catálogos base que usan los bienes

F-01 Alta prioridad
Gestión de datos maestros (Categoría, Ubicación, Responsable)
Permite a la app móvil crear y consultar los catálogos base del sistema. Sin estos datos no se puede registrar ningún bien. Los tres catálogos tienen operaciones CRUD básicas y son creados desde la API, no son valores fijos.

Criterios de aceptación generales

  • Cada entidad tiene su propio endpoint con rutas independientes.
  • No se permite crear registros con nombre vacío o duplicado dentro del mismo catálogo.
  • El JSON de respuesta usa API Resource con estructura consistente en todas las entidades.
  • Los registros que tienen bienes asociados no pueden eliminarse.
HU-01-01 Crear categoría Alta · MVP
Como operador de la app móvil, quiero crear una nueva categoría para poder clasificar los bienes al momento de registrarlos.
Dado que envío POST /api/v1/categorias con { "nombre": "Tecnología" }
Cuando el nombre no existe previamente en la tabla
Entonces la API retorna 201 con el objeto creado { id, nombre, created_at }
Dado que envío POST /api/v1/categorias con { "nombre": "Tecnología" } nuevamente
Entonces la API retorna 422 con error "El nombre ya existe"
Dado que envío POST /api/v1/categorias con nombre vacío
Entonces la API retorna 422 con error de validación en el campo nombre
Notas técnicas: Validación: required|string|max:255|unique:categorias,nombre. Ruta: routes/api/v1/categorias.php. Controller delgado → CategoriaService → Categoria::create().

Feature F-02

Registro de bienes

CRUD principal del sistema — crear, listar, ver y editar activos físicos

F-02 Alta prioridad
Registro y gestión de bienes (activos físicos)
Núcleo del sistema. Permite registrar cada activo físico con su etiqueta única auto-generada, editarlo y consultarlo. Cada bien está vinculado a una categoría obligatoria y opcionalmente a un responsable y una ubicación.

Criterios de aceptación generales

  • La etiqueta se genera automáticamente en formato B00001..B99999. Nunca la ingresa el usuario.
  • No se puede crear un bien con una categoría inexistente o dada de baja.
  • El JSON de respuesta siempre incluye los datos anidados de categoría, responsable y ubicación.
  • El listado soporta paginación (15 por página por defecto).
HU-02-01 Crear bien con etiqueta auto-generada Alta · MVP
Como operador de la app móvil, quiero registrar un nuevo bien para que quede en el inventario con su etiqueta única y sus datos básicos.
Dado que envío POST /api/v1/bienes con { "nombre": "Laptop Dell XPS", "categoria_id": 1, "descripcion": "Core i7, 16GB" }
Cuando los datos son válidos y la categoría existe
Entonces la API retorna 201 con el bien creado incluyendo etiqueta generada (ej: B00042)
Y el estado del bien es "activo" por defecto
Dado que dos requests llegan simultáneamente
Entonces las etiquetas generadas son distintas y sin duplicado (protegido por DB::transaction)
Notas técnicas: La etiqueta se genera en BienService dentro de DB::transaction: 'B' . str_pad(Bien::max('id') + 1, 5, '0', STR_PAD_LEFT). Estado inicial: 'activo'. categoria_id: required|exists:categorias,id. responsable_id y ubicacion_id: nullable.
HU-02-02 Listar bienes con paginación Alta · MVP
Como operador de la app móvil, quiero ver el listado paginado de bienes activos para poder revisar el inventario completo.
Cuando envío GET /api/v1/bienes
Entonces recibo 200 con los primeros 15 bienes activos (no dados de baja)
Y cada bien incluye: id, etiqueta, nombre, estado, categoria, ubicacion, responsable
Y la respuesta incluye metadatos de paginación: total, per_page, current_page, last_page
Cuando envío GET /api/v1/bienes?page=2
Entonces recibo los bienes del 16 al 30
Notas técnicas: Usar ->paginate(15). Solo bienes donde deleted_at IS NULL (SoftDeletes automático). Eager loading: with(['categoria', 'responsable', 'ubicacion']).
HU-02-03 Ver detalle de un bien Alta · MVP
Como operador de la app móvil, quiero ver todos los datos de un bien específico para conocer su estado, ubicación y responsable actual.
Dado que existe un bien con id = 42
Cuando envío GET /api/v1/bienes/42
Entonces recibo 200 con todos los campos del bien más sus relaciones completas
Dado que el bien con id = 99 no existe
Entonces la API retorna 404 con mensaje "Bien no encontrado"

HU-02-04 Editar datos de un bien Alta · MVP
Como operador de la app móvil, quiero actualizar el nombre, descripción o categoría de un bien registrado para corregir o ampliar su información.
Dado que envío PUT /api/v1/bienes/42 con { "nombre": "Laptop Dell XPS 15", "categoria_id": 1 }

Dado que el bien está dado de baja (deleted_at no null)

Feature F-03

Control de estado y baja de bienes

Ciclo de vida del bien: activo, en reparación, en desuso, dado de baja

F-03 Alta prioridad
Control de estado y soft delete
Gestiona el ciclo de vida de cada bien. Los estados válidos son: activo, en_reparacion, en_desuso. La baja usa soft delete — el registro permanece en la BD con deleted_at pero no aparece en listados normales. Un bien dado de baja no puede ser editado ni reasignado.

Criterios de aceptación generales

  • Los únicos estados válidos son: activo, en_reparacion, en_desuso. Cualquier otro retorna 422.
  • Un bien dado de baja no aparece en el listado general de bienes activos.
  • La baja no borra el registro — establece deleted_at y estado = "dado_de_baja".
  • El motivo de baja es un campo de texto libre, obligatorio al dar de baja.
HU-03-01 Cambiar el estado de un bien Alta · MVP
Como operador de la app móvil, quiero cambiar el estado de un bien para reflejar si está activo, en reparación o en desuso.
Dado que existe el bien B00042 con estado "activo"
Cuando envío PATCH /api/v1/bienes/42/estado con { "estado": "en_reparacion" }
Entonces la API retorna 200 con el bien actualizado y estado = "en_reparacion"
Cuando envío estado = "perdido" (valor inválido)
Entonces la API retorna 422 con error "Estado no válido"
Notas técnicas: Endpoint dedicado PATCH /bienes/{id}/estado. Validación: in:activo,en_reparacion,en_desuso. No usar el endpoint PUT general para cambios de estado — separación de responsabilidades. Ruta: Route::patch('/bienes/{bien}/estado', ...).
HU-03-02 Dar de baja un bien (soft delete) Alta · MVP
Como operador de la app móvil, quiero dar de baja un bien indicando el motivo para registrar que fue retirado del inventario sin perder el historial.
Dado que existe el bien B00042 con estado "activo"
Cuando envío DELETE /api/v1/bienes/42 con { "motivo_baja": "Equipo obsoleto, no reparable" }
Entonces la API retorna 200 con { "message": "Bien dado de baja correctamente" }
Y el bien ya no aparece en GET /api/v1/bienes
Y el campo deleted_at queda con fecha y hora del momento de la baja
Cuando envío DELETE sin motivo_baja
Entonces la API retorna 422 con error "El motivo de baja es obligatorio"
Notas técnicas: Usar SoftDeletes trait en el Model Bien. El campo motivo_baja se guarda antes de llamar $bien->delete(). Se puede implementar con un FormRequest DeleteBienRequest que valide solo motivo_baja. El DELETE no borra — ejecuta $bien->delete() que setea deleted_at.
HU-03-03 Listar bienes dados de baja Media
Como operador de la app móvil, quiero consultar el listado de bienes dados de baja para tener un registro histórico de activos retirados.
Cuando envío GET /api/v1/bienes/dados-de-baja
Entonces recibo 200 con listado paginado de bienes donde deleted_at IS NOT NULL
Y cada bien incluye: etiqueta, nombre, motivo_baja, deleted_at (fecha de baja)
Notas técnicas: Usar Bien::onlyTrashed()->paginate(15). Ruta independiente antes del apiResource para que no entre en conflicto con {id}.
Feature F-04

Asignación de bienes

Vincular un bien a un responsable y a una ubicación física

F-04 Alta prioridad
Asignación de responsable y ubicación
Resuelve la pregunta central del inventario: ¿quién tiene este bien y dónde está? La asignación es independiente de la edición general del bien — usa endpoints dedicados para mantener claridad en el historial de cambios y simplicidad en las validaciones.

Criterios de aceptación generales

  • Un bien dado de baja no puede ser asignado ni reasignado.
  • El responsable y la ubicación deben existir en sus respectivas tablas.
  • La asignación es opcional — un bien puede existir sin responsable o sin ubicación.
  • La respuesta siempre devuelve el bien completo con sus relaciones actualizadas.
HU-04-01 Asignar bien a un responsable Alta · MVP
Como operador de la app móvil, quiero asignar un bien a una persona responsable para saber quién tiene ese activo a su cargo.
Dado que existe el bien B00042 y el responsable con id = 5
Cuando envío PATCH /api/v1/bienes/42/responsable con { "responsable_id": 5 }
Entonces la API retorna 200 con el bien actualizado y responsable anidado en el JSON
Cuando envío responsable_id = 999 (no existe)
Entonces la API retorna 422 con "El responsable no existe"
Cuando envío { "responsable_id": null }
Entonces la API retorna 200 — el bien queda sin responsable asignado
Notas técnicas: Validación: nullable|integer|exists:responsables,id. Permite desasignar enviando null. Ruta: Route::patch('/bienes/{bien}/responsable', [BienController::class, 'asignarResponsable']).
HU-04-02 Asignar bien a una ubicación Alta · MVP
Como operador de la app móvil, quiero indicar en qué lugar físico se encuentra un bien para saber dónde encontrarlo.
Dado que existe el bien B00042 y la ubicación con id = 3
Cuando envío PATCH /api/v1/bienes/42/ubicacion con { "ubicacion_id": 3 }
Entonces la API retorna 200 con el bien actualizado y ubicacion anidada en el JSON
Cuando envío { "ubicacion_id": null }
Entonces la API retorna 200 — el bien queda sin ubicación asignada
Notas técnicas: Misma lógica que responsable. Ruta: Route::patch('/bienes/{bien}/ubicacion', ...). Validación: nullable|integer|exists:ubicaciones,id.
HU-04-03 Reasignar un bien (cambiar responsable o ubicación) Media
Como operador de la app móvil, quiero reasignar un bien que ya tiene responsable o ubicación para reflejar que cambió de manos o de lugar.

Feature F-05

Consulta y búsqueda de bienes

Encontrar bienes por etiqueta, categoría, responsable o ubicación

F-05 Media prioridad
Búsqueda y filtrado de bienes
Permite a la app móvil encontrar bienes específicos usando query parameters en el endpoint de listado. La búsqueda por etiqueta es el caso de uso más frecuente (escaneo de código). Los filtros son combinables entre sí.

Criterios de aceptación generales

  • Los filtros se aplican como query params sobre GET /api/v1/bienes.
  • Los filtros son acumulables: ?categoria_id=1&estado=activo funciona correctamente.
  • Si no hay resultados, retorna 200 con array vacío, nunca 404.
  • La paginación se mantiene en todos los casos de filtro.
HU-05-01 Buscar bien por etiqueta Alta · MVP
Como operador de la app móvil, quiero buscar un bien por su etiqueta (escaneada con la cámara) para acceder de inmediato a su información.

Entonces recibo 200 con el bien que tiene esa etiqueta exacta
Cuando envío GET /api/v1/bienes?etiqueta=B99999 (no existe)
Entonces recibo 200 con { "data": [] }
Notas técnicas: Filtro exacto (no LIKE): ->when($request->etiqueta, fn($q, $v) => $q->where('etiqueta', $v)). Se puede implementar con un scope local en el Model o en BienService. No crear endpoint separado — usar el listado con query param.

Notas técnicas: ->when($request->categoria_id, fn($q, $v) => $q->where('categoria_id', $v)). Validar que categoria_id sea integer si está presente.

Feature F-06

Resumen e indicadores del inventario

Endpoints de agregación para el dashboard de la app móvil

F-06 Baja prioridad
Indicadores y resúmenes agregados
Provee datos agregados para el dashboard inicial de la app móvil. Permite responder rápidamente preguntas de negocio: cuántos bienes hay, cuántos están en reparación, cuántos no tienen dueño asignado. Son endpoints de solo lectura (GET) sin parámetros.

Criterios de aceptación generales

  • Todos los endpoints de esta feature son GET de solo lectura.
  • Los conteos no incluyen bienes con soft delete.
  • La respuesta siempre es un JSON plano (sin paginación).
  • Tiempo de respuesta aceptable: menos de 300ms con hasta 10.000 bienes.
HU-06-01 Conteo de bienes por estado Media
Como operador de la app móvil, quiero ver un resumen de cuántos bienes hay en cada estado para tener una visión general del inventario al abrir la app.
Cuando envío GET /api/v1/bienes/resumen/estados
Entonces recibo 200 con { "activo": 142, "en_reparacion": 8, "en_desuso": 15, "total": 165 }
Notas técnicas: Bien::selectRaw('estado, count(*) as total')->groupBy('estado')->get(). Construir el array de respuesta en el Service. Ruta antes de apiResource para evitar conflicto con {bien}.
HU-06-02 Resumen de bienes por categoría Baja
Como operador de la app móvil, quiero ver cuántos bienes tiene cada categoría para entender la distribución del inventario.
Cuando envío GET /api/v1/bienes/resumen/categorias
Entonces recibo 200 con array de { "categoria": "Tecnología", "total": 87 } ordenado descendente
Notas técnicas: JOIN con categorias. Bien::with('categoria')->selectRaw('categoria_id, count(*) as total')->groupBy('categoria_id')->orderByDesc('total')->get().
HU-06-03 Bienes sin responsable asignado Baja
Como operador de la app móvil, quiero ver el listado de bienes que no tienen responsable para poder asignarlos y no perderlos.
Cuando envío GET /api/v1/bienes/sin-responsable
Entonces recibo 200 con listado paginado de bienes activos donde responsable_id IS NULL
Notas técnicas: Bien::whereNull('responsable_id')->paginate(15). Solo bienes activos (soft delete automático).
HU-06-04 Bienes sin ubicación asignada Baja
Como operador de la app móvil, quiero ver los bienes que no tienen ubicación registrada para poder localizarlos físicamente y actualizar su lugar.
Cuando envío GET /api/v1/bienes/sin-ubicacion
Entonces recibo 200 con listado paginado de bienes activos donde ubicacion_id IS NULL
Notas técnicas: Bien::whereNull('ubicacion_id')->paginate(15). Misma lógica que sin-responsable.
MVP completado cuando: Las 15 historias de prioridad Alta pasan todos sus criterios de aceptación en ambiente de staging. Las 9 historias Media/Baja constituyen el backlog del segundo sprint.
Decisiones de arquitectura que impactan todas las HU: Todas las rutas viven en routes/api/v1/. Todos los endpoints siguen el patrón Controller (delgado) → Service (lógica) → Model (BD). Todas las respuestas usan API Resources. Todas las validaciones usan FormRequest dedicados. Los soft deletes son transparentes gracias al trait SoftDeletes — el filtro es automático en todos los queries.
